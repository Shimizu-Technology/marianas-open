# Marianas Open — Translation Strategy

## Overview

The Marianas Open website supports 6 languages:
- **English (en)** — source language
- **Japanese (ja)**
- **Korean (ko)**
- **Chinese (zh)**
- **Tagalog/Filipino (tl)**
- **Portuguese (pt)**

Translation is handled at two distinct layers, each with its own mechanism:

| Layer | Content Type | Managed By | Translated By |
|-------|-------------|------------|---------------|
| **Static UI** | Navigation, buttons, headers, labels, page text | Locale JSON files (`web/src/locales/`) | General Translation CLI (build-time) |
| **Dynamic Content** | Event descriptions, schedules, prize names, hotel info, etc. | Admin panel → Database | Phase 2: Backend translation job (after-save) |

---

## Phase 1: Static UI Translation (General Translation CLI)

### What It Translates

All text defined in locale JSON files — approximately 560+ keys including:
- Navigation items (Home, Events, About, Rules, etc.)
- Section headers (EVENT SCHEDULE, TOTAL PRIZE POOL, etc.)
- Button labels (Register Now, Share, Cancel, Save, etc.)
- Page content (About page, Rules page, Terms page, etc.)
- Form labels and placeholders
- Error messages and status text

### How It Works

1. Developers write all UI text in English in `web/src/locales/en.json`
2. At build time, the GT CLI (`npx gt translate`) reads `en.json`
3. GT's AI translation engine generates accurate translations for all 5 target languages
4. The translated files (`ja.json`, `ko.json`, `zh.json`, `tl.json`, `pt.json`) are written in-place
5. The Vite build then bundles these files into the app as usual
6. At runtime, `react-i18next` loads the appropriate locale file based on user preference

### Configuration

**`web/gt.config.json`:**
```json
{
  "$schema": "https://assets.gtx.dev/config-schema.json",
  "defaultLocale": "en",
  "locales": ["ja", "ko", "zh", "tl", "pt"],
  "customMapping": {
    "tl": { "code": "fil", "name": "Filipino" }
  },
  "files": {
    "json": {
      "include": ["src/locales/[locale].json"]
    }
  }
}
```

> **Note:** Tagalog uses the locale code `tl` in our file names, but GT's BCP 47 support uses `fil` (Filipino). The `customMapping` tells GT to translate `tl.json` using the Filipino language model.

**Build pipeline (`web/package.json`):**
```json
{
  "scripts": {
    "translate": "npx gt translate",
    "translate:safe": "... sources .env then checks GT_API_KEY/GT_PROJECT_ID ...",
    "build": "npm run translate:safe && node scripts/generate-seo-files.mjs && tsc -b && vite build",
    "build:no-translate": "node scripts/generate-seo-files.mjs && tsc -b && vite build"
  }
}
```

**Environment variables:**
- **Local development**: Add `GT_API_KEY` and `GT_PROJECT_ID` to `web/.env`
- **Production (Netlify)**: Add them as Netlify environment variables
```
GT_API_KEY=gtx-api-xxxxx     # Production key from generaltranslation.com
GT_PROJECT_ID=prj_xxxxx      # Project ID from generaltranslation.com
```

### Activation Steps

1. Go to https://dash.generaltranslation.com/signup and create an account
2. Create a project and get your `GT_API_KEY` and `GT_PROJECT_ID`
3. Add them to `web/.env`:
   ```
   GT_API_KEY=gtx-api-your-key
   GT_PROJECT_ID=prj_your-project-id
   ```
4. Test locally: `cd web && npm run translate` — this translates all locale files
5. Verify: check that `ja.json`, `ko.json`, `zh.json`, `tl.json`, `pt.json` have been updated
6. Add the same `GT_API_KEY` and `GT_PROJECT_ID` to Netlify environment variables
7. Deploy — translations auto-generate on every build

### Translation Context (Metadata)

For ambiguous terms in the Marianas Open context, a companion metadata file
(`en.metadata.json`) can provide hints to improve translation accuracy:

```json
{
  "event": {
    "tripPackages": {
      "context": "Jiu-jitsu tournament prizes: winners receive funded trip packages to compete at the championship in Guam"
    },
    "asjjfStars": {
      "context": "ASJJF (Asia Sport Jiu-Jitsu Federation) star ranking for tournament prestige level"
    }
  }
}
```

### When Translations Run

- **Development**: Translations do NOT run. Developers work with whatever locale files are committed.
- **Production deploy (Netlify)**: `npx gt translate` runs as the first step of the build. GT detects changed keys in `en.json` and only re-translates those, keeping costs minimal.
- **Force retranslation**: Run `npx gt translate --force` to regenerate all translations from scratch.

### Cost

- **Starter ($5/mo)**: Includes $10 usage credit. Covers ~560 keys across 5 languages with frequent deploys. GT only re-translates changed keys, keeping costs minimal.

## Phase 2: Dynamic Content Translation (Future)

### What It Will Translate

All content entered by the Marianas Open admin team:
- Event descriptions, taglines, schedule notes
- Schedule item descriptions
- Prize category names
- Accommodation details (hotel names, descriptions, room types)
- Registration steps and fee labels
- Travel and visa item descriptions
- Sponsor descriptions (if added)

### Planned Architecture

```
  Admin saves event ──► Rails API ──► Save to DB (English)
                                          │
                                          ▼
                                   Background Job
                                   (ActiveJob + Sidekiq)
                                          │
                                          ▼
                                   Translation API
                                   (GT / OpenAI / Google)
                                          │
                                          ▼
                                   Save translations
                                   (JSONB column per record)
                                          │
                                          ▼
                                   API returns translations
                                   alongside original content
                                          │
                                          ▼
                                   Frontend checks user locale
                                   and displays translated version
```

### Storage Design (Option C — JSONB blob per record)

Each model with translatable content gets a `translations` JSONB column:

```ruby
# Example: Event model
# translations column stores:
{
  "description": {
    "ja": "マリアナス・プロシリーズがマニラに...",
    "ko": "마리아나스 프로 시리즈가 마닐라에...",
    "zh": "马里亚纳斯职业系列赛来到马尼拉...",
    "tl": "Ang Marianas Pro Series ay darating sa Manila...",
    "pt": "A Marianas Pro Series chega a Manila..."
  },
  "prize_title": {
    "ja": "グアムへの道を勝ち取れ！",
    "ko": "괌행 티켓을 쟁취하세요!",
    ...
  }
}
```

### Frontend Integration

```typescript
// Helper to get translated or fallback text
function useTranslatedField(record: any, field: string): string {
  const { i18n } = useTranslation();
  const locale = i18n.language;

  if (locale === 'en') return record[field];
  return record.translations?.[field]?.[locale] || record[field];
}
```

### Implementation Details (deferred to Phase 2)

- Background job fires after create/update of translatable records
- Translation runs asynchronously; English content is available immediately
- Translations appear within seconds (async job completes)
- Re-translation triggered when source field content changes
- Admin can see translation status per record (translated / pending / failed)
- Fallback: if translation is missing, display English original

## i18next Integration (Unchanged)

The existing `react-i18next` setup remains exactly as-is. GT only generates
the locale JSON files — it does not replace the i18n runtime. The app continues
to use:

- `useTranslation()` hook for accessing static translations
- `i18next-browser-languagedetector` for detecting user language preference
- `localStorage` key `i18nextLng` for persisting language choice
- `LanguageSwitcher` component for manual language selection

No changes to `web/src/i18n.ts` or any component's `useTranslation` usage are needed.
